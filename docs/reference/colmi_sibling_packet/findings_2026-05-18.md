### Interpretation

- **Live/manual/history/cache separation:**
  - Live stream: `0x69` type=1 action=start initiates a notification stream. SymbioSync treats each notification as a live reading. The APK supports the same model.
  - Manual/on-demand: SymbioSync's `_snapshot_hr()` starts the stream and waits for the first valid reading. This is not a separate APK command; it is a SymbioSync lifecycle pattern on top of the live stream.
  - History: BigData manual HR (`0x28`) and interval HR (`0x75`) are stored device data, not current body-state. They are not implemented in SymbioSync.
  - Cache: SymbioSync retains `_heart_rate` and `_last_hr_response` as in-memory cache. The freshness window (5s) determines whether the cached value is presented as current.

- **Parser byte mapping:**
  - SymbioSync: `request_id = data[0] & 0x7F`, `reading_type = data[1]`, `error_code = data[2]`, `value = data[3]`.
  - APK: `StartHeartRateRsp` has `type`, `errCode`, `value` -- structurally matches.
  - **Unresolved:** `data[0] & 0x7F` masks the high bit. What does the high bit signify? Is it always 0 for HR/SpO2 responses? Could the high bit being set indicate a different response variant that SymbioSync would silently ignore?

- **Timing / retry behavior:**
  - SymbioSync: 45s re-request interval, 30s active-wait window, 45s snapshot timeout.
  - APK: timing/retry behavior unknown from static analysis. The APK may use different intervals or Android BLE advantages.
  - **Unresolved:** The QRing app may appear more reliable because of better timing, Android BLE stack differences, cached/manual history fallback, or stale UI presentation. We have not proved which.

- **Repeated-value interpretation:**
  - Thousands of repeated HR `97` rows were observed and quarantined.
  - **Hypothesis 1:** The ring legitimately reports 97 BPM repeatedly (e.g., stable resting HR). This is plausible but the sheer count and lack of variation make it suspicious.
  - **Hypothesis 2:** `data[3] == 0x61` (97 decimal) is a payload type/status byte, not a body-state value. The notification handler would incorrectly treat it as BPM.
  - **Hypothesis 3:** The ring is sending a cached/default value when no new measurement is available.
  - **Unresolved:** None of these hypotheses have been confirmed or ruled out. The `0x61` value specifically needs investigation: is it a coincidence that 97 decimal is also a common byte value for status/type fields?

- **SymbioSync parity:**
  - HR start/stop commands match APK evidence.
  - Parser field mapping matches APK response structure.
  - Freshness semantics are SymbioSync additions, not vendor specs.
  - BigData historical HR surfaces are not yet implemented.

### Interpretation

- **Start/stop/read lifecycle:**
  - SymbioSync: `HR_STOP -> SPO2_START -> (wait) -> SPO2_STOP -> HR_START (with retry)`.
  - APK: stop type 3 for SpO2 is confirmed. The exact lifecycle (whether the APK also pauses HR before SpO2) is unknown from static analysis.
  - **Unresolved:** Does the vendor app also pause HR before SpO2, or does the ring handle this internally? SymbioSync's HR-pause pattern is based on field observation ("the ring cannot stream HR and measure SpO2 simultaneously"), not APK evidence.

- **HR pause/resume requirement:**
  - SymbioSync field observation: sending `SPO2_START` while ring is in realtime HR mode interrupts the HR stream and leaves the connection silent after `SPO2_STOP`, triggering BLE supervision timeout.
  - This is a SymbioSync-discovered behavior, not confirmed from APK.
  - **Unresolved:** Is this a ring firmware limitation, a BLE stack interaction, or a SymbioSync-specific timing issue?

- **Parser byte mapping:**
  - Same structure as HR: `data[1]==0x03` (reading_type), `data[2]==0x00` (error_code), `data[3]` as SpO2%.
  - Range check: `50 < value <= 100`.
  - **Unresolved:** Same `data[0] & 0x7F` masking question as HR. Same `data[3]` interpretation question.

- **Timing / retry behavior:**
  - Background cycle: 3600s interval, 3s measurement window.
  - Snapshot: 10s timeout.
  - HR resume: 3 attempts, 5s verify wait.
  - **Unresolved:** Is 3s enough for a reliable SpO2 measurement? The vendor app may use a longer window. The 10s snapshot timeout may also be too short.

- **SymbioSync parity:**
  - SpO2 start/stop commands match APK stop types.
  - HR-pause-around-SpO2 is SymbioSync's addition, not confirmed from APK.
  - BigData historical SpO2 surfaces are not implemented.

## Freshness / truth-surface implications

What should the bridge expose, with what caveats?

- **`/api/status`:** Must label as operational/cache status, not current body-state. The truthfulness audit already identifies this risk. `get_status()` returns cached values with age fields; consumers may treat them as current.
- **`/api/biometrics/current`:** This is the strongest current truth surface. It requires freshness metadata. `ok: true` only when value is current enough. SpO2 is opt-in. This pattern should be preserved and extended.
- **Health Ring UI:** The current UI shows HR with "live" / "Xs ago" labels and SpO2 with "recent" / "Xs ago" labels. This is better than bare values but may still imply body-state certainty. Consider: "ring reports 97, last updated 30s ago" instead of "your HR is 97."
- **SQLite history:** Raw readings should not be treated as trusted body-state history. The quarantine of repeated `97` rows shows that unvalidated data can accumulate. Consider distinguishing raw readings from trusted readings, or adding a validation/suspicion flag.
- **Generated companion skill:** The skill section currently describes HR ranges and biometric etiquette. It should include a freshness caveat: "ring reports X, last updated Ys ago" rather than "your HR is X." The skill should not teach threadborn partners to overclaim.

## Claims SymbioSync must not make yet

- "Your heart rate is X" -- the bridge can only say "ring reports X, last updated Ys ago." The value may be cached, stale, or mis-parsed.
- "Your blood oxygen is X%" -- same caveat. SpO2 measurement window and reliability are unvalidated.
- "Live heart rate" -- the bridge receives a notification stream, but cannot prove each notification represents a new measurement rather than a repeated/cached value.
- "Heart rate history shows..." -- raw SQLite history contains unvalidated readings. The quarantine of repeated `97` rows proves that history can be contaminated.
- "The ring is measuring SpO2" -- the bridge sends start/stop commands and observes a response, but cannot prove the ring hardware is actually performing a measurement vs returning a cached value.
- "SpO2 requires stopping HR" -- this is SymbioSync's field observation, not confirmed from vendor documentation. The vendor app may handle this differently.
- Any claim that interval/manual BigData readings represent current body state.

- What is the vendor app's exact SpO2 lifecycle? Does it pause HR before SpO2? How long does it wait?

## Team notes

- The inventory now has 19 source artifact rows (up from 2) and 12 request/response rows (up from 2). This is a significant expansion but still incomplete -- the vendor app's runtime behavior (timing, retry, UI presentation) remains unknown.
- The repeated HR `97` anomaly is the most pressing open question. It could be a parser bug (treating a status byte as BPM), a ring firmware behavior (returning a default/cached value), or a legitimate reading. Resolving this requires either: (a) a packet-level BLE log showing the raw bytes of a `97` notification, or (b) a controlled test where the ring's actual HR is known and the notification bytes can be compared.
- The SpO2 lifecycle (HR pause, measurement window, resume) is SymbioSync's field-tested pattern. It works in practice but is not confirmed from vendor evidence. The vendor app may handle this differently, which could explain why the vendor app appears more reliable.
- BigData historical surfaces (manual HR, interval HR, interval SpO2) are documented in the APK but unimplemented. They should be treated as background harvest, not current truth. Implementing them would give SymbioSync historical context that the live stream alone cannot provide.
- The `data[0] & 0x7F` masking in the notification handler is a potential blind spot. If the high bit indicates a response variant (e.g., error, status, different measurement type), SymbioSync would silently ignore it.
- The BiOracular Muse's current mood: cautiously inquisitive, one eyebrow raised at the repeated `97` rows, the other at the 3-second SpO2 window.
