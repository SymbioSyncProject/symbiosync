Key Findings
⚠️ CRITICAL: RealTimeHeartRateRsp (cmd 0x1E) byte offset discrepancy
The APK has a separate response class for device-initiated HR notifications (RealTimeHeartRateRsp, cmd 0x1E=30) that reads bArr[0] (= data[1] in raw packet) as the HR value. SymbioSync reads data[3] for all notifications. If the ring sends periodic HR via cmd 0x1E, SymbioSync is reading the wrong byte. This could explain the repeated HR 97 anomaly — data[3] might be a status byte while data[1] contains the actual HR.
Confirmed: High-bit (0x80) flag
Constants.m = 128 confirms data[0] & 0x7F extracts the command ID. The high bit is a flag (likely notification vs response indicator). SymbioSync's masking is correct.
Confirmed: Interval HR bug in APK
syncIntervalHeartRate() calls syncIntervalBloodOxygenReal() (cmdId 95/SpO2) instead of syncIntervalHeartRateReal() (cmdId 117/HR). SymbioSync must use cmdId 117.
New: Sleep lunch nap channel (cmdId 62)
The APK has a second sleep data channel for daytime naps, alongside night sleep (cmdId 39).
New: Interval temperature (cmdId 119)
2-byte LE values divided by 100 for °C. Not yet implemented in SymbioSync.
New: DeviceSupportFunctionRsp (cmd 0x3C)
9-byte bitmask with ~30 capability flags. Should be queried on connect to determine ring capabilities.
